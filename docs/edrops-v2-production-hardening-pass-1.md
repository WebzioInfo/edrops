# Edrops V2 Production Hardening Pass 1

Date: 2026-06-03

## Broken Features Found

- Customer cart checkout called `POST /cart/sync`, but no backend cart API existed.
- Customer checkout called `POST /checkout/initialize`, but the checkout controller had no route.
- Customer cart payment success only showed a toast and cleared local storage; it did not verify payment with the backend.
- Customer recharge called `POST /payment/create-order`, while backend payment code also used `create-intent` semantics.
- Customer recharge verified payment but did not call the recharge purchase API, so purchased jar packs were not credited.
- Customer delivery tracking called `/deliveries/history`, but backend route was `/delivery/...`.
- Customer support called `/support/tickets`, but no support API existed.
- Promo discounts were displayed during recharge, but recharge purchase validation still expected full package price.

## Buttons And Forms Fixed

- Cart `Proceed to Pay`
  - Syncs local cart into database.
  - Initializes backend checkout.
  - Loads Razorpay SDK.
  - Verifies payment with backend callback.
  - Confirms linked order after signature verification.
  - Clears cart after backend checkout order creation.

- Recharge `Secure Recharge Checkout`
  - Creates Razorpay payment order.
  - Verifies payment with backend callback.
  - Calls recharge purchase API after verified payment.
  - Credits jar balance through existing recharge engine.
  - Supports validated fixed/percentage promo discounts in purchase validation.

- Support `Open Ticket`
  - Persists real `SupportTicket` rows.
  - Loads authenticated customer ticket history.
  - Uses DTO validation for subject, description, and priority.

- Delivery tracker load action
  - Uses a real authenticated `/delivery/history` endpoint.
  - Returns UI-shaped delivery history from database delivery/report records.

## APIs Connected

- Added `CartModule`
  - `GET /cart`
  - `POST /cart/sync`
  - `DELETE /cart`

- Connected `CheckoutController`
  - `POST /checkout/initialize`

- Added `SupportModule`
  - `GET /support/tickets`
  - `POST /support/tickets`

- Added delivery customer history endpoint
  - `GET /delivery/history`

- Added payment alias for frontend compatibility
  - `POST /payment/create-order`

- Hardened payment verification
  - Verifies Razorpay signature.
  - Checks payment ownership.
  - Records payment attempt idempotently.
  - Marks payment `SUCCESS`.
  - Confirms linked order.
  - Writes `PaymentAuditLog`.

## Database Integration Performed

- Cart sync now writes `Cart` and `CartItem`.
- Checkout reads cart from database and creates `Order` and `OrderItem`.
- Checkout clears persisted cart items after order creation.
- Support form writes `SupportTicket`.
- Delivery history reads `Delivery` and `DeliveryReport`.
- Recharge completion writes `PackagePurchase`, updates `JarBalance`, writes `Transaction`, and creates notification through the existing recharge engine.
- Promo discount purchase validation reads `PromoCode`, `PromoCampaign`, and `PromoRedemption`, then redeems the promo after successful recharge.

## Dummy Features Removed

None removed in this pass. The priority was to connect visible customer actions that already existed in UI. Several admin/staff dashboards still need a later dummy/static-data sweep.

## Tables Removed

None. Database cleanup/removal was not performed in this pass because the active codebase still references legacy prepaid and newer commerce structures. Removing tables before completing the dependency map would be risky.

## Dead Code Removed

None in this pass.

## Performance Improvements

- Cart sync merges duplicate product lines before writing database rows.
- Cart product validation fetches only needed fields.
- Support tickets and cart reads use bounded list sizes.
- Delivery history returns a shaped payload instead of sending full customer/order relation graphs.

## Verification

- Backend TypeScript check passed.
- Frontend TypeScript build check passed.
- Backend production build passed.
- Frontend production build passed.

## Test Status

`npm run test` currently fails. The failures are from existing autogenerated Jest specs that instantiate controllers/services without their dependencies, for example CheckoutController without CheckoutService and PaymentService without PrismaService/RazorpayProvider. This is a test harness issue that should be fixed by replacing those generated smoke specs with proper provider mocks or module-level tests.

## Remaining Production Risks

- DB-backed granular RBAC is still not implemented across all controllers.
- Commerce checkout still lacks warehouse stock reservation and deduction.
- Money columns still use Float in Prisma schema.
- Refunds, failed payment handling, invoice generation, and webhook retry/reconciliation are incomplete.
- Admin/staff pages still need full button/form audit and permission enforcement.
- Legacy prepaid inventory and V2 warehouse stock are still split.
